# Bookiverse

Bookiverse는 도서 대출 시스템을 여러 서비스로 나누어 구현하는 프로젝트다.

하나의 대출 요청이 도서 조회, 대출 생성, 재고 차감, 결과 반영, 연체, 반납, 포인트 처리로 이어질 때 각 서비스가 어떤 데이터를 소유하고 어떻게 협력하는지 확인하는 데 목적이 있다.

## 프로젝트 목적

이 프로젝트는 도서 대출이라는 익숙한 도메인으로 다음 흐름을 끝까지 다뤄보는 것을 목표로 한다.

- 사용자가 책을 대출한다.
- 대출 서비스는 도서 정보를 조회하고 대출 요청을 접수한다.
- 도서 서비스는 재고를 차감하고 결과를 대출 서비스에 전달한다.
- 대출 서비스는 재고 차감 결과에 따라 대출을 확정하거나 실패 처리한다.
- 반납일이 지나면 대출을 연체 상태로 전환한다.
- 연체 도서를 반납하면 대출 제한을 일부 해제하되, 남은 연체료는 포인트로 갚게 한다.

이 과정에서 서비스 간 데이터 공유, 이벤트 기반 상태 변경, 중복 이벤트 처리, 연체 정책, 포인트 정산 흐름을 함께 검증한다.

## 서비스 구조

| 서비스 | 책임 |
| --- | --- |
| `gateway` | 외부 요청 진입점 |
| `book` | 도서 정보, 재고, 재고 차감 결과 |
| `book-catalog` | 도서 카탈로그 영역 실험 |
| `rental` | 대출 상태, 반납, 연체, 대출 제한 정책 |
| `user` | 사용자 정보와 포인트를 소유할 경계 |

각 서비스는 자신의 데이터를 직접 소유한다. 다른 서비스의 DB를 직접 조회하거나 수정하지 않고, 필요한 정보는 API 조회나 이벤트로 주고받는다.

## 전체 데이터 흐름

```text
client -> gateway -> rental
                 rental -> book: 도서 정보 조회
                 rental -> rental DB: RentalItem(PENDING) 저장
                 rental -> Kafka: StockDeductRequested
book -> book DB: 재고 차감
book -> Kafka: StockDeducted / StockDeductFailed
rental -> rental DB: RENTED 또는 FAILED 반영
```

대출 요청 직후에는 재고 차감 결과가 아직 없기 때문에 `RentalItem`을 `PENDING`으로 둔다. 이후 `book`의 결과 이벤트를 받아 `RENTED` 또는 `FAILED`로 확정한다.

```text
PENDING -> RENTED | FAILED
RENTED -> OVERDUE -> RETURNED
```

## 유스케이스

| 상태 | 유스케이스 | 설명 |
| --- | --- | --- |
| 구현됨 | 도서 조회 | `rental`이 대출 요청 시 `book`에서 도서 정보를 조회한다. |
| 구현됨 | 대출 요청 접수 | `rental`이 대출 가능 여부를 확인하고 `RentalItem(PENDING)`을 만든다. |
| 구현됨 | 재고 차감 요청 | `rental`이 `StockDeductRequested` 이벤트를 outbox에 저장한다. |
| 구현됨 | 재고 차감 처리 | `book`이 이벤트를 소비해 재고를 차감하고 결과 이벤트를 발행한다. |
| 구현됨 | 대출 확정/실패 | `rental`이 결과 이벤트를 받아 `RENTED` 또는 `FAILED`로 변경한다. |
| 구현됨 | 연체 처리 | scheduler가 반납일이 지난 대출 항목을 `OVERDUE`로 변경한다. |
| 구현됨 | 연체 대출 제한 | 미반납 연체 도서가 있으면 대출 불가, 연체료만 남으면 제한 대출 상태로 둔다. |
| 구현됨 | 반납 처리 | `rental`이 대출 항목을 `RETURNED`로 변경한다. |
| 예정 | 반납 재고 복구 | 반납 완료 후 `book`에 재고 복구 이벤트를 전달한다. |
| 예정 | 사용자 포인트 | `user` 서비스가 포인트 원장을 소유하고, `rental`은 포인트 포트로 연동한다. |
| 예정 | 포인트 상계 | 대출로 생기는 포인트를 연체료 상환에 먼저 사용하고 남은 포인트만 적립한다. |
| 예정 | Kafka retry/DLT | 기술 실패를 retry와 DLT/DLQ로 분리해 처리한다. |

## 연체 정책

`rental`은 책을 돌려주지 않은 상태와 책은 돌려줬지만 연체료가 남은 상태를 구분한다.

- 미반납 연체 도서가 있으면 대출 불가
- 연체 도서를 반납했지만 연체료가 남아 있으면 제한 대출
- 제한 대출 상태에서는 1권만 대출 가능
- 대출로 생기는 포인트는 연체료 상환에 먼저 사용

연체료는 정책으로 계산하며, 현재 기본 정책은 하루 100P, 최대 3일 부과다.

## 신뢰성

이벤트 발행은 outbox 패턴을 사용한다.

- 도메인 상태 변경과 outbox 저장을 같은 트랜잭션에서 처리한다.
- 별도 publisher가 outbox record를 Kafka로 발행한다.
- 재고 차감은 요청 ID 기준 처리 이력을 남겨 중복 소비에 대응한다.

재고 부족 같은 비즈니스 실패는 실패 이벤트로 처리하고, Kafka 소비 실패나 DB 장애 같은 기술 실패는 retry와 DLT/DLQ 대상으로 둔다.
